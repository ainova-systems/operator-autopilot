# Resolve `## Execution Summary` contract drift (supervisor / AOP agents)

## Problem
The success path of `FileAgentInvocation.invoke` extracts a summary from the
**action agent's** stdout by looking for a `## Execution Summary` block
(`engine/pipeline/primitives/agent-invocation.ts:66-68` â†’ `extractOrSynthesizeSummary`,
`:178-192`). The supervisor role prompt
(`engine/content/prompts/agents/supervisor.md:135-160`) never emits that block â€”
it carries its summary in the AOP `=== EMIT verdict ===` `summary:` field. So
every successful supervisor run logs the misleading
`WARN agent output missing ## Execution Summary block â€” using synthesized fallback`
(observed on PR #1) and falls back to a `[synthesized] verdict=approved. <500-char slice>`
summary.

That synthesized fallback then leaks:
- into KV execution history (`engine/pipeline/run-stage.ts:489` persists
  `agentResult.summary`), degrading reviewer/retrospective inputs; and
- **into a user-facing PR comment** on the no-changes/escalate path
  (`engine/pipeline/composers/pr-feedback-supervisor-stage.ts:509-513`), where
  the literal `[synthesized]` text is posted as the supervisor's reasoning.

The supervisor's real summary is already parsed into `applied.summary`
(`aop-applier.ts:258,295`), but the composer's two **success** paths
(`pr-feedback-supervisor-stage.ts:487-498` and `:499-519`) ignore it and use
`agentResult.summary`. The terminal paths (`:464`, `:483`) already override
with `applied.summary`.

## Decision (record it in code)
The `## Execution Summary` block is **OPTIONAL**, not required. AOP-transport
agents (supervisor, and any stage whose summary comes from an `EMIT verdict
summary:` field surfaced via the composer's `afterAgent` override) legitimately
omit it. The synthesized fallback is a **designed safety net**, not a prompt
defect â€” so its log line is DEBUG, and the supervisor prompt stays as-is (it
correctly uses AOP `summary:`).

## Solution
1. In `engine/pipeline/primitives/agent-invocation.ts` (`extractOrSynthesizeSummary`,
   `:185-192`): downgrade the `log?.warn(...)` for the missing-block/synthesized
   path to `log?.debug(...)`, and update the surrounding JSDoc/comment
   (`:165-171`, `:185-187`) to state the fallback is a designed path and the
   block is optional â€” remove the stale "find verifier prompts that still lack
   the block" framing. The `[synthesized]` marker on the returned summary is the
   retained visible diagnostic.
2. In `engine/pipeline/composers/pr-feedback-supervisor-stage.ts`: on the two
   approved success paths â€” fix-in-place-with-changes (`:487-498`) and
   no-changes/escalate (`:499-519`) â€” use `applied.summary` (falling back to
   `agentResult.summary` when empty) as the summary source. Return
   `{ summaryOverride: applied.summary || agentResult.summary }` (or equivalent)
   so KV history carries the supervisor's own words, and post
   `applied.summary || agentResult.summary` as the no-changes reasoning block
   instead of the raw `agentResult.summary`.

Keep both changes in this one task/PR â€” they are the same contract decision and
must revert together.

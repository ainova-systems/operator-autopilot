# Correct the engine layer graph for `infra/` and `logging/`

## Problem
The authoritative engine layer graph in `intelligence/rules/typescript.md`
(the single source of truth developers and analyzer agents use to judge
intra-engine import legality — it is NOT ESLint-enforced for intra-engine
layers) understates the real imports of its two lowest layers:

- `intelligence/rules/typescript.md:106` — `infra/ → Node built-ins only` is
  false. `infra/` imports `@operator/core` (runtime + types), `better-sqlite3`
  (under `infra/local/`), and one type from `logging/`.
- `intelligence/rules/typescript.md:107` — `logging/ → nothing` is false.
  `logging/` imports `pino`, `node:stream`, and `type OperationContext` from
  `@operator/core`.

This is docs↔code drift, which `intelligence/rules/dev-context-engineering.md`
defines as a defect ("documentation that drifts from code is a defect; fix it
in the same change"). An agent trusting the stale graph would flag these
legitimate imports as boundary violations, producing false findings and review
churn. Line 59 of the same file already records that error classes moved from
`engine/infra/errors.ts` into core — which is precisely why `infra/` now
depends on core — but the graph was never updated to match.

## Verified evidence (actual imports)
`infra/` (non-test):
- `engine/infra/env.ts:3` — `ConfigError` (runtime) from `@operator/core`
- `engine/infra/workspace.ts:5,7` — `OperationContext` (type) + `WorkspaceError` (runtime)
- `engine/infra/workspace-init.ts:5,6` — core types + `WorkspaceError` (runtime)
- `engine/infra/workspace-init.ts:7` — `type { Logger }` from `../logging/logger.js` (infra → logging, type-only)
- `engine/infra/instance-heartbeat.ts:3,4` — core types + `instanceEntrySchema` (runtime)
- `engine/infra/local/sqlite-guard.ts:2,3,4` — `better-sqlite3` (third-party) + core types

`logging/` (non-test):
- `engine/logging/logger.ts:1` — `pino` (third-party)
- `engine/logging/logger.ts:2` — `node:stream`
- `engine/logging/logger.ts:3` — `type OperationContext` from `@operator/core`

## Solution
Edit only `intelligence/rules/typescript.md` (doc-only; no code change — the
imports themselves are correct and architecturally sound: infra sits above
logging, logging sits above nothing, core is the base contract). Replace the
two graph lines so they read (matching the annotation style already used
elsewhere in the graph, e.g. "types only", "through primitives only"):

```
infra/                → @operator/core, logging/ (type Logger), Node built-ins (+ better-sqlite3 under infra/local/)
logging/              → @operator/core (types), pino, Node built-ins
```

Do NOT drop the `logging/ (type Logger)` on the `infra/` line — it is required
by the acceptance criterion below (`workspace-init.ts:7`). Keep line width and
alignment consistent with the surrounding graph block.

## Affected Files
- `intelligence/rules/typescript.md:106` — replace `infra/ → Node built-ins only`
- `intelligence/rules/typescript.md:107` — replace `logging/ → nothing`

## Acceptance Criteria
- `intelligence/rules/typescript.md` `infra/` line lists `@operator/core`,
  `logging/` (type), Node built-ins, and `better-sqlite3` (scoped to `infra/local/`)
- `intelligence/rules/typescript.md` `logging/` line lists `@operator/core` (types),
  `pino`, and Node built-ins
- A grep of `engine/infra/**` and `engine/logging/**` non-test imports shows no
  dependency outside what the updated graph permits
- No code files changed; imports are unmodified
- `npm run typecheck && npm run lint && npm test` pass (build unaffected by a doc-only edit)

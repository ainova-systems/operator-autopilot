# Give `GitHubVCS` a logger and paginate `getCheckRuns`

## Problem
Two defects in `engine/platforms/github/vcs.ts` on the CI-check I/O path:

1. **Silent swallow (`vcs.ts:386-404`).** `getCheckRuns` ends in a bare
   `catch { return []; }`. The `GitHubVCS` constructor (`vcs.ts:170-174`) takes
   only `octokit, owner, repo`, so this method has no logger and cannot warn —
   malformed responses and API errors vanish as "no checks".
2. **Non-paginated list (`vcs.ts:391-393`).** `checks.listForRef({ …, per_page: 100 })`
   is called directly, NOT via `this.octokit.paginate(...)`. A PR with >100 check
   runs truncates to page 1. If a failing check sits on page 2 while page 1 is all
   green, `aggregateChecks` (`observe-status.ts:292`) returns `"passing"` and the
   operator treats a red PR as green — a concrete "failing looks passing" bug on
   large monorepos. Every other list call in this file already paginates
   (`pulls.list`, `issues.listComments`, `issues.listLabelsOnIssue`,
   `checks.listAnnotations`, `git.listMatchingRefs`, `repos.listBranches`).

## Solution
1. Add a constructor-injected logger to `GitHubVCS`, consistent with the
   composition root:
   - Import `Logger` from `../../logging/logger.js` (layer-legal — `auth.ts:2`
     already does this).
   - Add the logger as a constructor parameter (`vcs.ts:170-174`). Prefer making it
     optional (`logger?: Logger`) so existing test construction at `vcs.test.ts:74`
     and any other callers stay valid; the composition root always supplies one.
   - Pass `logger` at all three instantiation sites in `engine/entry.ts` (`:221`,
     `:326`, `:370`) — the `logger` variable is already in scope at each (it is
     passed to `createOctokit`).
2. Replace the bare `catch` at `vcs.ts:401` with a `catch (err)` that emits
   `this.logger?.warn(...)` carrying `codeReviewId`, the error message, and `.cause`
   before returning `[]`.
3. Replace the direct `checks.listForRef` call at `vcs.ts:391` with
   `this.octokit.paginate(this.octokit.rest.checks.listForRef, { owner, repo, ref: pr.head.sha, per_page: 100 })`.
   `paginate` flattens `check_runs` across pages; map the flattened array through
   `mapCheckRun` as today (drop the `data.check_runs` indirection).

## Observability
- Adds one **WARN** in `getCheckRuns` on the catch-and-continue path with
  `codeReviewId` + error reason + `.cause`. Pagination is a pure correctness fix
  (no visibility change) and preserves the existing return contract (`CheckRun[]`).

## Affected Files
- `engine/platforms/github/vcs.ts:170-174` — add `logger?: Logger` constructor param
  + `import type { Logger }`.
- `engine/platforms/github/vcs.ts:391-393` — swap direct `listForRef` for
  `this.octokit.paginate(...)`, flatten `check_runs`.
- `engine/platforms/github/vcs.ts:401-403` — WARN with `codeReviewId` + `.cause`
  before `return []`.
- `engine/entry.ts:221, 326, 370` — pass `logger` into `new GitHubVCS(...)`.
- `engine/platforms/github/vcs.test.ts:74` — thread a fake/stub logger; add the
  regression tests below. Follow the existing `paginate` mock pattern already used
  by the `getComments` / `getLabels` tests in this file.

## Acceptance Criteria
- Pagination regression test: a fixture with >100 check runs where the failing run
  is on page 2 returns every run across pages (including the page-2 failure), and
  feeding the result through `aggregateChecks` yields `"failing"`. The test must
  fail on the pre-fix single-page code.
- WARN regression test: when the underlying Octokit call rejects, `getCheckRuns`
  returns `[]` and the injected logger's `warn` is called with `codeReviewId` and
  the error cause; fails on pre-fix code (which logs nothing).
- No push / PR / branch behavior changes; constructor change is additive and each
  `entry.ts` call site compiles.
- `npm run typecheck && npm test && npm run lint` pass (coverage ≥90% on
  `vcs.ts`; no `ts-prune`/`knip` regressions from the signature change).

# Give the `failed` queue-fill outcome its own throttle + backoff semantics

## Problem
In `runProject`, the `queue-fill` post-run block at
`engine/engine/project-runner.ts:113-123` only handles two outcomes:
- `completed` → `markScheduleRun` + `updateQueueFillBackoff`
- `skipped` (non-`locked`) → `markScheduleRun`

There is no branch for `result.status === "failed"`. `safeExecute`
(`project-runner.ts:245-247`) converts any thrown stage error into
`{ status: "failed" }`, so this path is reachable whenever the queue-fill
stage (research — an LLM/agent stage) genuinely fails.

Because `markScheduleRun` never runs on failure, `isScheduleDue`'s
base-interval throttle (`project-runner.ts:213`, keyed on
`schedule.stateKey`) stays satisfied every subsequent cycle. Because
`updateQueueFillBackoff` (`project-runner.ts:135-150`) is invoked only for
`completed`, the exponential-backoff counter (`schedule.backoffStateKey`)
is never bumped either. Net effect: a persistently failing research stage
re-fires on **every** cycle, defeating both the base interval and the
exponential backoff and burning agent budget on a broken stage. Failures
are treated more leniently than empty successes — the inverse of correct.

The existing tests (`project-runner.test.ts:454-518`) cover
completed-reset, completed-bump, locked-skip no-op, and no-eligible-skip
throttle-only — there is no `failed` case, confirming the gap is untested,
not intentional.

## Solution
Add an explicit `result.status === "failed"` branch inside the
`entry.schedule.kind === "queue-fill"` block in `runProject`
(`engine/engine/project-runner.ts:113-123`) that:
- calls `markScheduleRun(entry, project.id, deps, ctx)` to advance the
  base-interval throttle, and
- bumps the backoff counter **unconditionally** — a failure is never
  evidence the stage "produced something", so it must NOT go through
  `updateQueueFillBackoff`'s `hasInFlightOutput` reset path (which would
  reset the counter to `0` when an in-flight PR exists, e.g. under
  `skipScheduleCheck`). Increment `schedule.backoffStateKey` directly,
  mirroring the empty-run bump at `project-runner.ts:146-149`
  (`getCounter` → `setCounter(min(empty + 1, MAX_BACKOFF_EXPONENT))`), or
  extend `updateQueueFillBackoff` with an explicit force-bump mode used by
  the failed branch. Prefer whichever keeps `updateQueueFillBackoff`'s
  completed-run semantics intact.

Keep the lock-skip case a full no-op (unchanged). Update the comment block
at `project-runner.ts:103-112` to document the `failed` semantics alongside
completed / skipped-locked / skipped-other so a failure reads as
"throttle like an empty run" rather than falling through silently.

If a shared helper is extracted for the unconditional bump, keep it a named,
colocated, `ts-prune`/`knip`-clean function reachable from the run loop.

## Affected Files
- `engine/engine/project-runner.ts:113-123` — add the `failed` branch
  (advance throttle + unconditional backoff bump); keep lock-skip a no-op.
- `engine/engine/project-runner.ts:103-112` — extend the by-result comment
  block to document the `failed` outcome.
- `engine/engine/project-runner.test.ts:454-518` — add the failing-first
  regression test in the `runProject — queue-fill backoff` suite.

## Acceptance Criteria
- New regression test in `project-runner.test.ts` named for the bug (e.g.
  "a failed queue-fill run advances the throttle and backoff so it does not
  re-fire every cycle") that drives `executeAction` to resolve
  `{ action: "research", status: "failed" }` and asserts BOTH
  `state.markScheduleRun` is called with `(…, "sample", "research")` and
  `state.setCounter` is called with `(…, "sample", "research-empty", 1)`
  (counter starting at `0`). The test fails on current code and passes after
  the fix.
- A second assertion (or existing locked-skip test left green) confirms the
  lock-skip case remains a full no-op: neither `markScheduleRun` nor
  `setCounter` fires on `{ status: "skipped", message: "locked" }`.
- `runProject` handles `failed` explicitly in the queue-fill branch; the
  failed backoff bump never resets the counter to `0`.
- `npm run typecheck && npm run lint && npm test` pass (coverage on the
  touched file stays >=90%; `ts-prune` + `knip` clean).

# Replace the `node:fs/promises` mock in `runtime.test.ts` with a real temp dir and assert the temp-file lifecycle

## Problem
`engine/agents/runtime.test.ts:17-24` mocks `node:fs/promises`
(`writeFile`, `unlink`) to no-ops, so the agent system-prompt temp-file
lifecycle is never exercised. `AgentRuntime.run` writes the prompt to
`join(tmpdir(), "operator-prompt-...md")` (`runtime.ts:187-188`) and removes it
in a `finally` block (`runtime.ts:420`). No test verifies the file is actually
created, nor that it is cleaned up on the success **or** the failure path — a
temp-file leak on the error path would go undetected. This violates
`intelligence/rules/typescript.md` §REQUIRED ("Real temp directories for
filesystem tests via `fs.mkdtemp`. Never mock `fs`").

## Solution
Remove the `vi.mock("node:fs/promises", ...)` block entirely and let
`writeFile`/`unlink` run against a real, test-controlled temp directory.
Because the source hardcodes `tmpdir()` for the prompt-file location, redirect
it by mocking `node:os`'s `tmpdir()` to a per-test `fs.mkdtemp` directory
(mocking `os` is permitted; mocking `fs` is not) — or an equivalent mechanism
that keeps `fs` real. Follow the temp-dir lifecycle in
`engine/infra/workspace-init.test.ts:38-40` (`mkdtemp` in `beforeEach`, `rm` in
`afterEach`).

Assert the lifecycle against the real directory:
- **Created during the run**: have the fake `AgentProvider.execute` snapshot
  the temp directory (or the `systemFile` it receives) while the run is in
  flight and confirm exactly one prompt file exists — the write at
  `runtime.ts:188` happens before the provider executes.
- **Cleaned up on success**: after a successful `run`, assert the temp
  directory contains no leftover `operator-prompt-*` file (`finally` unlink ran).
- **Cleaned up on the failure path**: after a run that exhausts retries and
  rejects (e.g. provider returns `exitCode: 1` with `maxRetries: 1`), assert the
  temp directory is likewise empty — pinning the `finally` cleanup on the error
  path.

Name the new test(s) after the behavior they pin (e.g. "writes and removes the
system-prompt temp file on success" / "removes the system-prompt temp file even
when the run fails") so a future regression trips the same assertion.
Keep the existing `node:child_process` and `./prompt-builder.js` mocks.

## Affected Files
- `engine/agents/runtime.test.ts:17-24` — remove the `node:fs/promises` mock
- `engine/agents/runtime.test.ts` (top) — add real temp-dir lifecycle
  (`mkdtemp`/`rm`) and redirect `os.tmpdir()` to it
- `engine/agents/runtime.test.ts` (`describe("AgentRuntime")`) — add
  success-path and failure-path assertions for temp-file creation and cleanup,
  reading the real directory

## Acceptance Criteria
- No `vi.mock("node:fs/promises")` (or any `fs` mock) remains in
  `engine/agents/runtime.test.ts`
- A test asserts the system-prompt temp file is created (observed during the
  run) and removed after a successful run, against a real `fs.mkdtemp` dir
- A test asserts the temp file is removed on the failure path (retries
  exhausted) — no leaked prompt file
- Temp dirs are removed in `afterEach` (no leaked directories)
- `npm run typecheck && npm run lint && npm test` pass; coverage on
  `engine/agents/runtime.ts` stays >=90%

# Strip the deployment-configured token var from every agent subprocess env

## Scope
This work item is the **implementation** track for finding
[F20260705-F595B0A3](../findings/F20260705-F595B0A3.md). The parent finding PR
(`ai/findings/F20260705-F595B0A3`) documents the issue and spawns this task;
the code/test changes listed below land in the child task's implementation PR,
not in the finding PR itself.

## Problem
`buildChildEnv` (`engine/agents/providers/cli.ts:60-80`) copies the entire
operator `process.env` into every agent CLI subprocess, then deletes only the
four hardcoded names in `AGENT_FORBIDDEN_ENV` (`cli.ts:41-46`:
`GH_TOKEN`, `GITHUB_TOKEN`, `GH_ENTERPRISE_TOKEN`, `GITHUB_ENTERPRISE_TOKEN`).

The operator does NOT hold its GitHub credential under any of those names. It
reads the VCS token from the deployment-configured `project.vcs.tokenEnvVar`
at `engine/entry.ts:218`, `entry.ts:323`, and `entry.ts:367`; the tracker path
at `entry.ts:621` uses `project.tracker.tokenEnvVar || project.vcs.tokenEnvVar`.
The canonical VCS value is `MANAGED_REPO_GH_TOKEN` (`config/repos.yaml.example:23`,
`.env.local.example:7`). That variable is present in the operator process env
(Octokit authenticates with it at `entry.ts:218-219`) and is NOT in the denylist,
so the raw write-scoped token is copied verbatim into every agent subprocess.

A prompt-injected or misbehaving agent (fed untrusted issue text, PR comments,
CI logs, repo code) can then `GH_TOKEN="$MANAGED_REPO_GH_TOKEN" gh pr edit …`
or call the GitHub API directly to mutate the orchestrator-owned PR, merge/close
reviews, or push to any branch the token reaches — the exact hard-boundary
violation the guard exists to prevent, and the same class as the 2026-06-19
PR-description overwrite the guard's own test cites.

The existing regression test (`cli.test.ts:512-528`) only asserts the four
hardcoded names, so it passes green while the real credential leaks.

## Solution
Stop relying on a static denylist that cannot know the deployment's token
variable. Thread the configured token-var name(s) into the strip step from the
composition root (the one place that holds repo config), avoiding a wide change
across every `AgentRunInput` builder.

1. `engine/agents/providers/cli.ts`
   - Extend `buildChildEnv` to also strip a caller-supplied set of extra
     forbidden var names (e.g. add a `forbiddenExtra: readonly string[] = []`
     parameter) alongside `AGENT_FORBIDDEN_ENV`. Keep the empty-string/undefined
     filtering and override semantics unchanged.
   - Give `CLIAgentProvider` those extra names via its constructor (e.g. a new
     `forbiddenEnvVars: readonly string[]` param) and pass them through at the
     `buildChildEnv(process.env, options.env, this.forbiddenEnvVars)` call site
     (`cli.ts:150`).
   - Update the JSDoc at `cli.ts:30-39` and `cli.ts:57-59` to state the
     deployment-configured token var(s) are stripped in addition to the four
     standard GitHub names.
   - (Optional hardening, not required for AC) consider inverting to an explicit
     allowlist so any future token name is denied by default; if done, keep it
     in the same PR and prove it with tests.
2. `engine/entry.ts` (composition root, `entry.ts:241-246`)
   - Compute the de-duplicated union of `project.vcs.tokenEnvVar` and
     `project.tracker?.tokenEnvVar` across `config.repos`, and pass it into each
     `new CLIAgentProvider(providerId, providerConfig, logger, <union>)`.
3. `engine/agents/providers/cli.test.ts`
   - Add the regression test described below; keep the existing four-name test.

## Affected Files

## Acceptance Criteria
- The built agent subprocess env excludes the deployment-configured
  `tokenEnvVar` value (e.g. `MANAGED_REPO_GH_TOKEN`) in addition to the four
  standard GitHub names.
- Regression test (MUST fail on pre-fix code, name describes the bug scenario):
  given a parent env containing `MANAGED_REPO_GH_TOKEN` and a configured
  forbidden set of `["MANAGED_REPO_GH_TOKEN"]`, the built agent env does not
  contain that key. Include a provider/entry-level assertion that a
  config whose `vcs.tokenEnvVar` is `MANAGED_REPO_GH_TOKEN` results in that key
  being absent from the child env.
- The four existing hardcoded names remain stripped (existing test still green).
- JSDoc at `cli.ts:30-39` / `cli.ts:57-59` names the configured token var(s) as stripped.
- `>=90%` coverage on touched files.
- `npm run typecheck && npm run lint && npm test` all pass.

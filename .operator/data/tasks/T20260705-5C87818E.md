# Remove retired "Operator V3" branding and make package.json the version source of truth

## Problem
`engine/cli.ts` brands a v5 engine as two generations old:
- `engine/cli.ts:2` — module JSDoc: `CLI argument parser for Operator V3.`
- `engine/cli.ts:96` — `printHelp` first line: `"Operator V3 — AI-powered SDLC engine"`

Every operator running `--help` is told they use a retired product. The
existing version marker `operatorEngineVersion` in `engine/index.ts:1` is
hardcoded to `"5.0.0"`, which itself disagrees with `package.json` (`0.5.0`),
so simply reusing it would still not match `package.json`. Additional retired
labels survive in comments (`engine/events/types.ts:47,51`,
`engine/logging/logger.ts:156`).

## Solution
Make `package.json` the single source of truth for the shown version and drop
the generation label from the banner.

1. `engine/index.ts` — derive `operatorEngineVersion` from `package.json`
   instead of the hardcoded `"5.0.0"`. Resolve `package.json` at runtime from
   `import.meta.url` following the established pattern in
   `engine/infra/content-path.ts:52-53`
   (`dirname(fileURLToPath(import.meta.url))`, then walk up to the repo root),
   OR use a static `import ... with { type: "json" }` — `resolveJsonModule`
   is already enabled in `tsconfig.json:16`. Pick whichever keeps `typecheck`,
   `ts-prune`, and `knip` green; watch the `rootDir: "."` / `declaration`
   emit gotcha with a static JSON import. Keep `printHelp` a pure string
   builder (no `OperationContext` needed).
2. `engine/cli.ts:96` — render the banner from the product name + version,
   e.g. `` `Operator ${operatorEngineVersion} — AI-powered SDLC engine` ``
   (no `V3`/`V4`). Import `operatorEngineVersion` from `./index.js`.
3. `engine/cli.ts:2` — update the JSDoc header to drop `V3`
   (e.g. `CLI argument parser for the Operator engine.`).
4. `engine/events/types.ts:47,51` and `engine/logging/logger.ts:156` — remove
   the retired `V3` mentions from the comments, keeping the technical content.
5. Update tests: `engine/cli.test.ts:104` must stop asserting
   `"Operator V3"` and instead pin the banner to the `package.json` version
   (assert the version substring and assert the banner contains no
   `"V3"`/`"V4"`); `engine/index.test.ts:7` must assert
   `operatorEngineVersion` equals the `package.json` version rather than the
   hardcoded `"5.0.0"`.

Note: `entry.ts` consumes `operatorEngineVersion` (`--status` at
`entry.ts:155/167/528`, startup log at `entry.ts:606`), so its version output
updates automatically — no edit to `entry.ts` required, and the startup INFO
log line is preserved. Leave the current-generation `"Operator v5 starting"`
literal at `entry.ts:606` as-is (out of scope: it is the current, not a
retired, label).

## Affected Files
- `engine/index.ts:1` — derive `operatorEngineVersion` from `package.json`
- `engine/index.test.ts:7` — assert version matches `package.json`, not `"5.0.0"`
- `engine/cli.ts:2` — drop `V3` from module JSDoc
- `engine/cli.ts:96` — render banner from name/version, no generation label
- `engine/cli.test.ts:104` — regression test pinning banner to package.json version + asserting no `V3`/`V4`
- `engine/events/types.ts:47,51` — remove `V3 new` from comments
- `engine/logging/logger.ts:156` — remove `V3` mention from comment

## Acceptance Criteria
- `printHelp()` output contains no `V3`/`V4` generation label and shows a
  version equal to `package.json`'s `version`.
- No `Operator V3` / `Operator V4` (or bare `V3`/`V4` generation) strings
  remain in `engine/`, `packages/`, or `app/` live source (historical docs
  excluded).
- A regression test (named for the bug scenario, e.g. "help banner shows the
  package.json version, not a retired generation label") pins the banner to
  the `package.json` version and fails on the pre-fix `"Operator V3"` literal.
- `engine/index.test.ts` asserts `operatorEngineVersion` matches `package.json`.
- `npm run typecheck && npm run lint && npm test` all pass (ts-prune + knip clean).

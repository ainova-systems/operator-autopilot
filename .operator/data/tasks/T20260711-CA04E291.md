# Log each AOP parse diagnostic per-record in `applyAgentEvents`

## Problem
`parseAgentOutput` (`engine/pipeline/primitives/agent-output-protocol.ts`)
drops every malformed/invalid EMIT block via `continue`, attaching a detailed
diagnostic (`code`, `line`, `emitType`, `message`) for each — `validation-failed`
(`:189-201`), `yaml-parse-error` (`:158-165`), `payload-not-mapping` (`:176-185`),
`block-unclosed` (`:122-131`), `raw-frontmatter-leak` (`:95-104`), and
`unknown-emit-type` (warning, `:132-140`).

`applyAgentEvents` (`engine/pipeline/primitives/aop-applier.ts:143-326`) captures
these into `parseResult.diagnostics` (`:149-150`) but **never logs them
individually**. It logs applied child-items (INFO `:189`), apply-errors (ERROR
`:275`), and error-events (WARN `:239`) — but the parse diagnostics only become an
aggregate count in `summaryParts` (`:284-289`). Both callers confirm the gap:
`aop-planner-stage.ts:440-447` logs `parse-errors=<count>` (count only; structured
payload omits even that), and `pr-feedback-supervisor-stage.ts:388-395` logs no
diagnostics at all.

Result: when a discovered finding/task fails Zod validation, the operator sees
`verdict=failed, parse-errors=1` with no way to tell **which** item was dropped or
**why** without re-running and grepping raw agent stdout — precisely the silent
data-loss the v5 observability mandate forbids (`.claude/rules/typescript.md`
§REQUIRED: "every catch-and-continue branch MUST warn with the reason"; "Silent
success is a bug").

## Solution
In `applyAgentEvents`, iterate `parseResult.diagnostics` and emit **one log line
per diagnostic**, keyed on `severity`:
- `severity === "error"` → `deps.log?.error(...)`
- `severity === "warning"` → `deps.log?.warn(...)`

Each line must include the diagnostic's `code`, `emitType`, `line`, and `message`
(both in the human-readable message and in the structured payload object). Suggested
placement: immediately after `partitionDiagnostics` (`aop-applier.ts:150`), before
the application loop, so dropped records are logged ahead of applied ones. Keep the
behavior non-throwing / best-effort — use the already-present optional `deps.log`
(`AopApplierDeps.log`, `:81`); do not change the verdict logic or the returned
`AopApplyResult` contract.

`AgentEventDiagnostic` shape (from `packages/core/src/types/agent-event-stream.ts:30-36`):
`{ severity, code, line, emitType?, message }`. Suggested structured payload keys:
`{ scope: "aop-applier", diagnosticCode: d.code, emitType: d.emitType, line: d.line }`.

Update the primitive's JSDoc "does NOT do" / forwarding note (`aop-applier.ts:30-38`)
only if needed to reflect that diagnostics are now surfaced at INFO-stream level.

Note on line cap: `aop-applier.ts` is under the `engine/pipeline/**` 200-line
code-cap (its length is mostly JSDoc/comments/log statements, which are excluded).
The added loop is a handful of code lines — verify it still fits after the change;
if it genuinely pushes the code-only count over, the loop is small enough to keep
inline, but do not strip logging to fit.

## Solution — regression test
Add a failing-first test to `engine/pipeline/primitives/aop-applier.test.ts` named
for the bug scenario (e.g. `"logs a per-record ERROR for a dropped validation-failed
child-item — parse diagnostics are no longer an invisible aggregate count"`).
- Build deps with a fake `Logger` (object with `error`/`warn`/`info`/`debug` as
  `vi.fn()`); inject via `deps.log`.
- Drive `fakeStream` with a `diagnostics` array containing a `validation-failed`
  error diagnostic (`{ severity: "error", code: "validation-failed", line: N,
  emitType: "child-item", message: "EMIT child-item validation failed — ..." }`)
  and zero events.
- Assert `deps.log.error` was called with a message/payload carrying `code`,
  `emitType`, `line`, and `message`.
- Add a companion assertion that a `warning`-severity diagnostic (e.g.
  `unknown-emit-type`) routes to `deps.log.warn`, not `error`.
- This test MUST fail on pre-fix code (no per-diagnostic log call) and pass after.

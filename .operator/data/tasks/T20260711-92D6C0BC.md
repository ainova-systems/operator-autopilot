# Make `parseAnalyzerFrontmatter` CRLF-tolerant + pin the contract

## Problem
`engine/pipeline/primitives/discovery-selector.ts:80-90` parses analyzer
frontmatter with the exact **pre-fix** logic that was already corrected in the
parallel parser: it splits on `block.split("\n")` and matches each line with
`/^(\w[\w_]*):\s*(.*)$/` (no `m` flag). On a managed-repo workspace checked out
on Windows / with git `autocrlf`, analyzer files (`.operator/analyst/*.md`)
carry `\r\n`. A bare `\n` split leaves a trailing `\r` on every value line, and
`(.*)$` (no `m` flag) cannot anchor across that `\r`, so **every field silently
fails to parse**.

The silent fallbacks in `loadAnalyzerDefs` (`discovery-selector.ts:118-125`)
then corrupt behavior:
- `schedule: fm.schedule || "daily"` → a `weekly`/`on-demand` analyzer runs
  EVERY cycle (wasted agent spend + duplicate findings).
- `enabled: fm.enabled !== "false"` → an `enabled: false` analyzer RUNS.
- `path: fm.path` → `undefined`, so analyzer scope scans the whole repo.

This is the identical contract already fixed in
`engine/work-items/work-items.ts:1081-1091` (`block.split(/\r?\n/)`) and pinned
by `work-items.test.ts:164` ("parses CRLF frontmatter ... (regression)"). The
`discovery-selector.ts` copy — a primitive (>=95% coverage bar) — was left
unpinned: `discovery-selector.test.ts` (287 lines) contains no CRLF fixture;
every fixture, including the `enabled: false` cases, uses `\n` only.

## Solution
1. **Add a failing-first, consumer-side CRLF regression test** to
   `engine/pipeline/primitives/discovery-selector.test.ts`, inside the
   `describe("loadAnalyzerDefs", ...)` block (alongside the existing
   `enabled: false` case at `:125`). Write an analyzer file whose bytes use
   `\r\n` line endings — e.g.
   `---\r\nschedule: weekly:3\r\nenabled: false\r\npath: Source/**\r\n---\r\n\r\nBody`
   — then call `loadAnalyzerDefs(analystDir)` and assert on the resulting
   `AnalyzerDef`:
   - `enabled === false` (stays disabled — fails on pre-fix code, which yields `true`),
   - `schedule === "weekly:3"` (stays throttled — fails on pre-fix code, which yields `"daily"`),
   - `path === "Source/**"` (preserved — fails on pre-fix code, which yields `undefined`).
   Name it for the bug scenario, e.g.
   `"parses CRLF (\\r\\n) analyzer frontmatter — enabled:false stays disabled and weekly:3 stays throttled (regression)"`.
   Confirm it FAILS on the current `split("\n")` parser before applying the fix.
2. **Fix the parser**: change `discovery-selector.ts:83` from
   `block.split("\n")` to `block.split(/\r?\n/)` so `schedule`, `enabled`, and
   `path` parse identically under LF and CRLF. Update the `:80` comment so it
   no longer claims to keep the (now-superseded) v4 semantics verbatim; note
   the CRLF tolerance, mirroring the rationale documented at
   `work-items.ts:1083-1090`.

Keep the change to these two files only so it is independently revertible in a
single commit. Do NOT bundle the broader "extract one shared frontmatter-field
parser across the copies" refactor into this task — that crosses a layer
boundary (`pipeline/primitives/` vs `work-items/`) and multiple call sites, and
belongs in its own scoped change if pursued. Mention it only as a follow-up
note in the PR description.

# Fix prompt ↔ code drift on global-context auto-detect order

## Problem
Bundled agent prompts document the global-context auto-detect priority in an
order that contradicts the engine's real behavior, and nothing in CI catches
the drift.

- `engine/content/prompts/agents/scout.md:53` embeds this comment inside the
  generated `project.yaml` template:
  `# Global context file (auto-detected if not set: CLAUDE.md > AGENTS.md > .cursorrules)`
  — the order is inverted (`CLAUDE.md` first) and the `.operator/OPERATOR.md`
  fallback is missing.
- `engine/config/discovery.ts:59-65` (`detectGlobalContext`) actually resolves
  in order `AGENTS.md → CLAUDE.md → .cursorrules → .operator/OPERATOR.md`
  (JSDoc `discovery.ts:42-46` and tests `discovery.test.ts:67-94` confirm this
  is the intended behavior).
- The false comment already propagated into generated output (self-init PR #1
  shipped the wrong order, caught only in human review). Every future onboarding
  reproduces it.
- `scout.md:68`, `analyst.md:22`, `creator.md:21`, and `context/base.md:17`
  list the same files CLAUDE-first and omit `.operator/OPERATOR.md`.

## Solution
Make `discovery.ts` the single source of truth for the order, correct every
prompt to match it, and add a contract test that fails CI on the next drift.

1. In `engine/config/discovery.ts`, extract the ordered candidate list into a
   named exported constant (e.g. `GLOBAL_CONTEXT_ORDER = ["AGENTS.md",
   "CLAUDE.md", ".cursorrules", ".operator/OPERATOR.md"]`) and rebuild the
   `candidates` array inside `detectGlobalContext` from it (map
   `.operator/OPERATOR.md` → `resolve(automationDir, "OPERATOR.md")`, the rest
   → `resolve(workspaceRoot, name)`). The constant must stay reachable from the
   runtime resolver AND the new test so `ts-prune`/`knip` see it consumed.
2. Fix `scout.md:53` to read the correct order including the fallback, e.g.
   `AGENTS.md > CLAUDE.md > .cursorrules > .operator/OPERATOR.md`, and keep it
   machine-parseable (single `>`-delimited list after a fixed prefix like
   `auto-detected if not set:`).
3. Align the ambiguous listings so they read `AGENTS.md, CLAUDE.md, .cursorrules,
   .operator/OPERATOR.md` (or at minimum stop implying CLAUDE-first and stop
   omitting `.operator/OPERATOR.md`): `scout.md:68`, `analyst.md:22`,
   `creator.md:21`, `context/base.md:17`.
4. Add a contract/regression test in `engine/config/discovery.test.ts` that:
   - loads the bundled `scout.md` via
     `resolveContentPath("prompts", "agents/scout.md")`
     (`engine/infra/content-path.ts:31`),
   - parses the documented `>`-delimited auto-detect order from the template
     comment,
   - asserts it deep-equals the exported `GLOBAL_CONTEXT_ORDER` from
     `discovery.ts`.
   Name it after the bug scenario (e.g.
   `scout.md documents the same global-context order that discovery.ts implements`).
   It MUST fail on the pre-fix `scout.md:53` text and pass after the fix.

## Affected Files
- `engine/config/discovery.ts:59-65` — export the ordered constant; rebuild `candidates` from it (update JSDoc `:42-46` if wording changes).
- `engine/content/prompts/agents/scout.md:53` — correct the priority comment + add `.operator/OPERATOR.md`.
- `engine/content/prompts/agents/scout.md:68` — align the order-ambiguous listing.
- `engine/content/prompts/agents/analyst.md:22` — align the listing.
- `engine/content/prompts/agents/creator.md:21` — align the listing.
- `engine/content/prompts/agents/context/base.md:17` — align the listing.
- `engine/config/discovery.test.ts` — add the contract test binding scout.md's documented order to the exported constant.

## Acceptance Criteria
- `scout.md:53` documents the auto-detect order exactly as implemented in `discovery.ts`, including the `.operator/OPERATOR.md` fallback.
- The four related listings (`scout.md:68`, `analyst.md:22`, `creator.md:21`, `context/base.md:17`) no longer imply CLAUDE-first and no longer omit `.operator/OPERATOR.md`.
- A test in `discovery.test.ts` fails if the documented order in `scout.md` and the implemented candidate list diverge; it fails on pre-fix `scout.md` and passes after the fix.
- No new dead code: the exported order constant is consumed by both the resolver and the test (`ts-prune`/`knip` clean).
- `npm run typecheck && npm run lint && npm test` pass; coverage on `engine/config/discovery.ts` stays >=90%.

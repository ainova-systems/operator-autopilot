# Demote expected-404 Octokit request-log lines from ERROR to DEBUG

## Problem
During normal label transitions the cycle log shows lines like
`ERROR GET /repos/.../labels/ai%3Aprocessing - 404`, immediately followed by
a successful label creation. The 404 is the expected "label does not exist
yet, create it" branch of `GitHubVCS.ensureLabel`, but it surfaces on the
ERROR stream. This violates the v5 observability tenet ("every ERROR line
represents a real failure"; `migration.md`) and trains operators to ignore
ERROR — which is how real failures get missed.

### Real mechanism (do NOT patch `ensureLabel`)
The ERROR line is **not** emitted by `ensureLabel` — that method logs nothing.
It is emitted by the built-in `@octokit/plugin-request-log`, which logs every
failed HTTP request via `octokit.log.error(...)`
(`node_modules/@octokit/plugin-request-log/dist-src/index.js:16`), and routed
straight to ERROR by the `error:` adapter in `createOctokit`
(`engine/platforms/github/auth.ts:19`). The exception is thrown AFTER this log
line, so the caller's `if (isNotFound(err))` catch cannot suppress it. Adding a
DEBUG log to `ensureLabel` would NOT remove the ERROR line.

Because every existence probe shares this one client, the fix belongs at the
`createOctokit` logging boundary and fixes all probes at once:
`getCodeReview` (`vcs.ts:210`), `removeLabel` (`vcs.ts:373`), `deleteBranch`
(`vcs.ts:499`), `getBranchTipCommitTime` (`vcs.ts:529`), `ensureLabel`
(`vcs.ts:546`), `getWorkItem` (`tracker.ts:66`).

## Solution
In `engine/platforms/github/auth.ts`, make the `error:` log adapter in
`createOctokit` status-aware. The request-log format is stable:
`"${method} ${path} - ${status} with id ${requestId} in ${ms}ms"`. Parse the
status token (e.g. match ` - (\d{3}) ` / ` - (\d{3})$`) and:
- route request-log lines whose status is `404` to `logger.debug` (expected
  "resource absent" — the benign outcome every probe treats as control flow);
- route everything else (non-404 statuses, and any message that does not match
  the request-log format) to `logger.error`, unchanged.

This loses no real signal: a 404 a caller does NOT catch still propagates as a
thrown exception and is logged at ERROR (with `.cause`) by the caller/stage
error path. Only the redundant transport-level echo of an expected 404 is
demoted. Keep the change small and centralized — do not add per-callsite logs
in `vcs.ts` / `tracker.ts`.

Add a short comment explaining WHY the demotion exists (expected existence
probes; ERROR reserved for real failures per the v5 observability tenet), so a
future reader does not "fix" it back.

## Affected Files
- `engine/platforms/github/auth.ts:14-21` — make the `error:` adapter in
  `createOctokit` classify request-log messages by status; demote 404 to
  `logger.debug`, keep non-404 and non-matching messages at `logger.error`.
- `engine/platforms/github/auth.test.ts` — add the regression test(s) below.

## Acceptance Criteria
- A request-log-formatted 404 message
  (`"GET /repos/o/r/labels/ai%3Aprocessing - 404 with id ABC in 12ms"`) routed
  through the `createOctokit` logger lands on `debug`, not `error`.
- A request-log-formatted non-404 failure (e.g. `- 500`, `- 403`) still routes
  to `error`.
- A plain non-request-log `error("...")` message still routes to `error` (the
  existing `auth.test.ts:22-39` behavior for `error:e` stays green).
- Regression test name describes the bug scenario (e.g.
  "routes an expected 404 request-log line to debug instead of error"), fails
  on pre-fix code (which routes all `error` messages to `logger.error`), and
  passes after the fix.
- `npm run typecheck && npm run lint && npm test` all pass; coverage on
  `auth.ts` stays ≥90%.

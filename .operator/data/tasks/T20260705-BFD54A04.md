# Retire the orphaned `dailyResearchHour` default; keep `daily` as a generic schedule kind

## Problem
`schedules.dailyResearchHour` is a research-specific engine default that no
production code reads. It is declared, validated, and threaded into
`DefaultsConfig`, but the production dispatch-registry builder
(`buildStageDispatchRegistryFromKV`) explicitly ignores `defaults`
(`engine/engine/kv-dispatch-registry.ts:76` — `void defaults`) and reads each
stage's schedule verbatim from its `kv:workflow-stages/*` row. The shipped
research stage uses `kind: queue-fill` (`engine/content/prompts/stages.yaml:74-83`),
not `daily`. The **only** reader of `schedules.dailyResearchHour` is the test
helper `engine/test-helpers/test-dispatch-registry.ts:66-72`, which fabricates
a `daily`-scheduled `research` stage — a config shape production never emits —
and whose own docstring falsely claims it "mirrors what
`buildStageDispatchRegistryFromKV` produces from the seeded stages.yaml." The
`defaults.yaml` comment ("UTC hour for daily research run") misleads operators
into believing research is clock-driven when it is queue-driven.

## Scope decision (read before editing)
This task retires ONLY the research-specific `dailyResearchHour` default and
corrects the stale test helper. It KEEPS the generic `daily` schedule kind:
`daily` is a first-class, config-selectable member of the `@operator/core`
`ScheduleSpec` contract (peer of `interval`/`weekly`/`queue-fill`), reachable
in production whenever any workflow-stage config selects
`dispatch.schedule.kind: daily` (via `stages.yaml` or the app config UI), and
it carries a direct evaluator unit test. Do NOT delete the `daily` variant, its
Zod schema member, or the `project-runner.ts` `daily` branch — that would remove
a supported feature from a public package contract.

## Solution
1. Remove the orphaned field from source:
   - `engine/content/defaults/defaults.yaml:6` — delete the
     `dailyResearchHour: 8 # UTC hour for daily research run` line.
   - `engine/config/loader.ts:30` — delete
     `dailyResearchHour: parsed.schedules.dailyResearchHour,`.
   - `packages/core/src/schemas/engine-defaults.schema.ts:47` — delete
     `dailyResearchHour: z.number().min(0).max(23),`.
   - `packages/core/src/types/config.ts:82` — delete
     `readonly dailyResearchHour: number;` from the schedules type.
2. Fix the stale test helper `engine/test-helpers/test-dispatch-registry.ts`:
   replace the `research` entry's `daily` schedule (sourced from the removed
   `schedules.dailyResearchHour`) with the `queue-fill` schedule matching
   production `stages.yaml` (order 70, `featureFlags: [dailyResearch]`,
   `kind: queue-fill`, `targetKind: finding`, `countStatuses: [pending, reopened]`,
   `target: 5`, `inFlightBranchPrefix: ai/research`, `baseIntervalMinutes: 120`,
   `maxBackoffMinutes: 10080`, `stateKey: research`, `backoffStateKey: research-empty`).
   Update the file's docstring so it no longer describes a daily research schedule.
3. Update every engine-defaults test fixture that sets `dailyResearchHour` so
   the suite still compiles (drop the key): `engine/config/loader.test.ts:38`,
   `engine/engine/project-runner.test.ts:61`, `engine/engine/kv-dispatch-registry.test.ts:9`,
   `engine/engine/engine.test.ts:26`, `engine/smoke.test.ts:133`,
   `engine/pipeline/composers/discovery-iteration-stage.test.ts:41`,
   `engine/storage/seed.test.ts:99` (YAML fixture string line),
   `packages/core/src/schemas/index.test.ts:274`.
4. In `engine/engine/kv-dispatch-registry.test.ts`, the `RESEARCH_ROW` fixture
   (:44-54) and the pass-through test (:143-149) use a literal `daily` schedule
   (not the removed field, so they compile). Keep at least one test that a
   `daily`-scheduled workflow-stage row is passed through verbatim by
   `buildStageDispatchRegistryFromKV` (this proves `daily` stays supported), but
   stop labelling that row "research" — rename it to a neutral/generic stage name
   to avoid re-implying research is daily.

  helper.

## Notes
- ~13 files, but this is one atomic change: once the schema/type field is gone,
  the helper and every fixture must move together or the build breaks. It is a
  single logical contract narrowing, revertible in one commit — do not split.
- No observability change: this touches config/type/schema/tests only, adds and
  removes no I/O log lines.

## Acceptance Criteria
- `dailyResearchHour` is removed from `defaults.yaml`, `loader.ts`,
  `engine-defaults.schema.ts`, `config.ts`, and every test fixture.
- The generic `daily` schedule kind (ScheduleSpec, workflow-stage schema,
  project-runner branch) is retained and still covered by a direct unit test.
- No runtime schedule branch is exercised exclusively by a test helper: the
  `research` entry in `test-dispatch-registry.ts` now uses `queue-fill` matching
  production, and the `daily` branch is covered by the direct `project-runner`
  evaluator test.
- A schema/config test pins the retired contract (defaults without
  `dailyResearchHour` parse; field absent from parsed result) and fails on
  pre-fix code.
- `npm run typecheck && npm run lint && npm test` pass (ts-prune/knip clean).

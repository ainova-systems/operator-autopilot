# Log a WARN in `observeChecks` before degrading CI status to "none"

## Problem
`engine/pipeline/primitives/observe-status.ts:334` — `observeChecks` wraps
`getCheckRuns` in `try { … } catch { return { value: "none", … } }`. The catch
discards the error even though `deps.log` is already in scope (it is passed to
`classifyChecksFailureMode` at line 330). A transient GitHub 5xx / network error
makes the operator record CI as `"none"` — indistinguishable from a repo that
genuinely has no checks — with zero log output. This is exactly the "I can't see
what the engine did" failure the v5 observability BLOCKER exists to eliminate.

The existing test at `observe-status.test.ts:335` ("treats getCheckRuns rejection
as a non-fatal 'none' observation") pins the swallow but asserts no log, codifying
the blind spot.

## Solution
In the `catch` block at `observe-status.ts:334`, before returning the `"none"`
degradation, call `deps.log?.warn(...)` with the caught error's message and
`.cause` (and the `prNumber` for correlation). Bind the caught error, e.g.
`catch (err) { deps.log?.warn(...); return { value: "none", observedAt, checks: [] }; }`.
Keep the return shape and the best-effort semantics (never throw) unchanged — this
adds visibility only.

## Observability
- Adds one **WARN** on the catch-and-continue path carrying the PR number, the
  error message, and `.cause`. No INFO/ERROR changes; the non-fatal degradation
  stays non-fatal.

## Affected Files
- `engine/pipeline/primitives/observe-status.ts:334` — bind the error in the
  catch and emit `deps.log?.warn` with reason + `.cause` before returning `"none"`.
- `engine/pipeline/primitives/observe-status.test.ts:335` — update the existing
  "treats getCheckRuns rejection as a non-fatal 'none' observation" test (or add a
  sibling) to pass a fake/`vi.fn()` logger via `deps.log` and assert `warn` was
  called with the error reason/cause. This is the regression test — it must fail on
  the pre-fix code (which logs nothing) and pass after.

## Acceptance Criteria
- `observeChecks` regression test: when `getCheckRuns` rejects, `deps.log.warn` is
  called with the error reason/cause; the test fails on pre-fix code.
- Existing behavior preserved: the observation still returns `{ value: "none", checks: [] }`
  and never throws.
- `npm run typecheck && npm test && npm run lint` pass (coverage ≥95% on the touched
  primitive `observe-status.ts`).

**Severity**: high
**Priority**: 2
**Files Affected**: 1 (root cause) + 2 caller sites

**Pattern**: Silent data loss on agent paths â€” "an agent-produced work item can be dropped without a log line" (Orchestration Reliability Rules Â§Work-item lifecycle integrity, P1-P2). The v5 observability mandate (`intelligence/rules/typescript.md` Â§REQUIRED) forbids a silent `continue` on a dropped agent record: every dropped/skipped EMIT must surface as WARN/ERROR + execution-log entry.
**Domain**: engine/pipeline/primitives (aop-applier.ts, agent-output-protocol.ts)

**Impact**: `parseAgentOutput` drops every malformed/invalid EMIT block via `continue`, attaching a **detailed** diagnostic (which block, which field, why): `validation-failed` (`agent-output-protocol.ts:189-201`), `raw-frontmatter-leak` (`:95-104`), `yaml-parse-error` (`:158-165`), `payload-not-mapping` (`:176-185`), `block-unclosed` (`:122-131`), and `unknown-emit-type` (warning, `:132-140`). A dropped `EMIT child-item` is an agent-discovered finding/task that never reaches `deps.source.create`.

`applyAgentEvents` (`aop-applier.ts:143-326`) captures these into `parseResult.diagnostics` (`:149-150`) but **never logs them individually**. It logs applied child-items (INFO `:189`), apply-errors that fail at `source.create` (ERROR `:275`), and error-events (WARN `:239`) â€” but the parse diagnostics only become an aggregate string count in `summaryParts` (`:284-289`). The applier's own doc (`:30-38`) states callers forward `notes`/`errors`/`recoveries` to `ExecutionHistory` â€” **diagnostics are not in that list**, so they also never reach `execution-events`. Both callers confirm the gap: `aop-planner-stage.ts:440-447` logs `parse-errors=<count>` (count only, and the structured payload omits even that), and `pr-feedback-supervisor-stage.ts:388-395` logs no diagnostics at all.

Result: when a discovered finding/task fails Zod validation, the operator sees `verdict=failed, parse-errors=1` with **no way to tell which item was dropped or why** without re-running and grepping raw agent stdout â€” precisely the "I can't see what the engine did" failure v5 exists to prevent. For a research run producing several valid findings plus one malformed one, the valid ones are created, the stage reports failed, and the lost finding's identity + validation reason are invisible.

**Fix**: In `applyAgentEvents`, before/after the application loop, iterate `parseResult.diagnostics` and emit one log line per diagnostic keyed on `severity` (ERROR for `severity === "error"`, WARN for `"warning"`), including `code`, `emitType`, `line`, and `message`. This makes every dropped/skipped EMIT record individually auditable and forwardable to the execution-history sink. Use the already-present `deps.log`; keep the non-throwing best-effort behavior.

**Severity**: medium
**Priority**: 4
**Files Affected**: 4

**Pattern**: Dead configuration and a production-unreachable schedule branch left behind by the queue-fill migration
**Domain**: engine/config, engine/engine, engine/content/defaults

**Impact**: `engine/content/defaults/defaults.yaml` still ships `schedules.dailyResearchHour` (commented "UTC hour for daily research run") and `engine/config/loader.ts` threads it into runtime config, but no stage consumes it: `engine/content/prompts/stages.yaml` contains no `kind: daily` schedule (research moved to `kind: queue-fill`), so the `daily` branch in `engine/engine/project-runner.ts` (hourUtc comparison) is only ever exercised by the test helper `engine/test-helpers/test-dispatch-registry.ts`, which fabricates a daily schedule from this very config. This misleads operators into believing research is clock-driven, keeps a production-unreachable branch alive via tests (masking the no-dead-code gate), and contradicts the queue-driven scheduling model.

**Fix**: Decide the contract: if the `daily` schedule kind is retired, remove `dailyResearchHour` from `defaults.yaml`, the loader, the `daily` branch in the project-runner schedule evaluator, and the test-helper fabrication in one expand-contract change; if it is kept as a supported stage-schedule kind for future stages, document that in `stages.yaml` comments and re-point the defaults comment away from research. Either way the test suite must stop being the only consumer of a runtime branch.

**Acceptance Criteria**:
- [ ] `dailyResearchHour` is either fully removed (config, loader, runner branch, test helper) or explicitly documented as a generic stage-schedule option not tied to research
- [ ] No runtime schedule branch is exercised exclusively by test helpers
- [ ] A schema/config test pins the decided contract
- [ ] Build passes after fix

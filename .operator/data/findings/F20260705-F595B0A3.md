**Severity**: high
**Priority**: 2
**Files Affected**: 1 (guard is `engine/agents/providers/cli.ts`; evidence spans runtime + config)

**Pattern**: Credential-boundary defeat — the "agent can never touch the GitHub credential / orchestrator-owned PR" control is a hardcoded 4-name denylist, but the operator's real write token lives under a deployment-configured, differently-named env var that the denylist never strips. Violates the base-context hard boundary "the orchestrator owns the pull request; the agent must never mutate it" and the Security Audit "tokens come from env vars referenced via `tokenEnvVar`" contract.
**Domain**: engine/agents/providers (agent subprocess environment construction)

**Impact**: `buildChildEnv` (`engine/agents/providers/cli.ts:60-80`) builds every agent subprocess env by copying the *entire* operator `process.env`, then deleting only the four hardcoded names in `AGENT_FORBIDDEN_ENV` (`cli.ts:41-46`: `GH_TOKEN`, `GITHUB_TOKEN`, `GH_ENTERPRISE_TOKEN`, `GITHUB_ENTERPRISE_TOKEN`) at `cli.ts:69`. It is invoked at `cli.ts:150` `env: buildChildEnv(process.env, options.env)`, and the sole caller `runtime.ts:264-271` passes **no** `env` override, so nothing else narrows the set.

But the operator does not hold its GitHub credential under any of those four names. It reads the VCS token via `process.env[project.vcs.tokenEnvVar]` at `engine/entry.ts:218`, `entry.ts:323`, and `entry.ts:367`; the tracker path at `entry.ts:621` uses `project.tracker.tokenEnvVar || project.vcs.tokenEnvVar`. The canonical VCS value in both `config/repos.yaml.example` (`tokenEnvVar: MANAGED_REPO_GH_TOKEN`) and `.env.local.example` (`MANAGED_REPO_GH_TOKEN=`) is **`MANAGED_REPO_GH_TOKEN`**. That variable is present in the operator process env (it is how Octokit authenticates) and is **not** in the denylist, so `buildChildEnv` copies the raw write-scoped token verbatim into every agent CLI subprocess.

**Remediation**: tracked in child task [T20260705-760BCFB4](../tasks/T20260705-760BCFB4.md); this finding PR documents the issue and spawns that task — code changes land in the task's implementation PR.

This is the exact incident the guard's own regression test was written for: `cli.test.ts:512-528` ("strips GitHub credential vars so the agent cannot authenticate gh against the PR", citing the 2026-06-19 case where an agent inherited an auth'd `gh` and overwrote the orchestrator-authored PR description). The test only asserts the four hardcoded names, so it passes while the real credential leaks. A prompt-injected or misbehaving agent (fed untrusted issue text, PR comments, CI logs, and repo code) can `GH_TOKEN="$MANAGED_REPO_GH_TOKEN" gh pr edit …` or call the GitHub API directly to mutate the orchestrator-owned PR, close/merge reviews, or push to any branch the token can reach — the precise hard-boundary violation the control exists to prevent.

**Fix**: Stop relying on a static denylist that cannot know the deployment's token variable. Thread the configured `tokenEnvVar` name(s) (`project.vcs.tokenEnvVar` and `project.tracker.tokenEnvVar`) into the agent-invocation path and strip them in `buildChildEnv` alongside `AGENT_FORBIDDEN_ENV` (e.g. accept an extra `forbiddenExtra: string[]` and delete those too). Preferably invert to an explicit allowlist of the vars an agent CLI actually needs (provider auth keys, PATH, HOME, cwd-related) so any future token name is denied by default. Update the JSDoc at `cli.ts:57-59` to state the configured token var is included.

**Acceptance Criteria**:
- [ ] The agent subprocess env excludes the deployment-configured `tokenEnvVar` value (e.g. `MANAGED_REPO_GH_TOKEN`) in addition to the four standard GitHub names
- [ ] Regression test (fails on pre-fix code): given a parent env containing `MANAGED_REPO_GH_TOKEN` and a config whose `tokenEnvVar` is `MANAGED_REPO_GH_TOKEN`, the built agent env does not contain that key
- [ ] Build passes after fix (`npm run typecheck && npm run lint && npm test`)

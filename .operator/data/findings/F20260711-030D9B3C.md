**Severity**: high
**Priority**: 2
**Files Affected**: 1 root cause (`engine/logging/logger.ts`) + every structured-data log call site; ready-made fix helper sits unused in `engine/logging/redact.ts`

**Pattern**: Secrets exposure in logs — external/sensitive values written to the log sink without redaction. Violates the zero-secret-leak expectation behind the redaction layer and the v5 observability mandate that DEBUG carry "HTTP response shapes … full payload dumps" (`intelligence/rules/typescript.md` §REQUIRED) — precisely the fields most likely to hold a token.
**Domain**: engine/logging (logger.ts, redact.ts)

**Impact**: `wrapPino` (`engine/logging/logger.ts:33-67`) applies `redactString` only to the message string and passes the structured `data` object straight to pino unmodified on all four levels (`:37`, `:44`, `:51`, `:58` — `pinoLogger.<level>(data, redactString(msg))`). The pino instances are constructed with **no** `redact` paths option (`:78-82`, `:139-140`, `:145`), so there is no pino-level fallback either. The file's own doc admits the gap: "secret redaction on messages" (`:19`, `:31`).

The codebase already built the correct tool for this: `redactValue` (`engine/logging/redact.ts:51-66`) recursively scrubs token patterns from nested objects/arrays, and its test explicitly pins that a `ghp_…` token nested inside a `data` object must become `[REDACTED]` (`engine/logging/redact.test.ts:83`, `:104`, `:116`). But `redactValue` is referenced **only** by its own definition and test — never by the logger or any runtime path — so the redaction the author intended for structured fields does not run.

Concrete leak path: on a non-zero agent exit, `engine/agents/providers/cli.ts:208-214` logs a WARN whose `data` carries `stderr` and `stdout` slices (1000 chars each) from the agent subprocess. Agent CLI failure output routinely echoes `Authorization: Bearer …` headers, auth-error dumps, or env values; all reach the log verbatim. That the sibling `engine/infra/workspace-init.ts:115-127` path hand-scrubs its subprocess output (`scrubbed`) confirms the team knows subprocess output needs scrubbing — the general logger simply doesn't do it for structured fields. Any token landing in a `data` field (error `.cause` strings, captured HTTP bodies, git remote URLs) is persisted in cleartext wherever logs are shipped or stored.

**Fix**: Wire `redactValue` into `wrapPino`: apply it to the `data` object (and to `child` bindings) before handing them to pino on every level, keeping the existing `redactString(msg)` for the message. Alternatively/additionally configure pino's native `redact` option. Update the "redaction on messages" doc comment to reflect that structured fields are now covered.

**Acceptance Criteria**:
- [ ] `wrapPino` passes every `data` object (and `child` bindings) through `redactValue` before pino serializes it, for debug/info/warn/error
- [ ] Failing-first regression test asserts a structured field containing a `ghp_…`/`sk-ant-…` token is emitted as `[REDACTED]` (fails on pre-fix logger, passes after), covering the `cli.ts` CLI-exit `stderr`/`stdout` case
- [ ] `redactValue` is reachable from the logger runtime, not only its test
- [ ] Coverage on touched files stays ≥90%
- [ ] Build passes after fix

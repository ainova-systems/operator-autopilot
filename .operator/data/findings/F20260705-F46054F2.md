**Severity**: high
**Priority**: 2
**Files Affected**: 2

**Pattern**: Silent recovery that swallows the original error (no WARN) on a safety-relevant boundary, plus a non-paginated GitHub list call ‚Äî violates the resilience rules "`catch {}` / a recovery that swallows the original error is a defect‚Ä¶ every recovery path must be logged" and "every list call paginates (`.paginate()`)", and the v5 observability BLOCKER "no silent success paths; every caught error gets a WARN/ERROR with `.cause`".
**Domain**: engine/pipeline/primitives/observe-status.ts + engine/platforms/github/vcs.ts (CI status observation ‚Üí verdict/merge decisions)

**Impact**: CI-check observation degrades silently at two stacked layers with zero log output.
- `engine/pipeline/primitives/observe-status.ts:334` ‚Äî `observeChecks` wraps `getCheckRuns` in `try { ‚Ä¶ } catch { return { value: "none", ‚Ä¶ } }`. `deps.log` is in scope (it is passed to `classifyChecksFailureMode` at line 330) yet the catch discards the error without a WARN. When a transient GitHub 5xx/network error hits `getCheckRuns`, the operator records CI as `"none"` ‚Äî indistinguishable from a repo that genuinely has no checks ‚Äî with no log line to diagnose. The existing test at `observe-status.test.ts:335` ("treats getCheckRuns rejection as a non-fatal 'none' observation") pins the swallow but asserts no log, codifying the blind spot. This is exactly the "I can't see what the engine did" failure v5 exists to eliminate, on the path that feeds status observation.
- `engine/platforms/github/vcs.ts:375` ‚Äî `getCheckRuns` itself has a bare `catch { return []; }` (no logger available on the class at all: `GitHubVCS` constructor takes only `octokit, owner, repo`, so this I/O method is neither `ctx`-threaded nor able to log), swallowing malformed-response and API errors as "no checks".
- `engine/platforms/github/vcs.ts:365` ‚Äî `checks.listForRef({ ‚Ä¶, per_page: 100 })` is fetched directly, NOT via `this.octokit.paginate(...)`. A PR with more than 100 check runs truncates to page 1. If a failing check sits on page 2 while page 1 is all green, `aggregateChecks` (observe-status.ts:292) returns `"passing"` and the operator treats a red PR as green ‚Äî a concrete "failing looks passing" correctness path on large monorepos.

**Fix**:
- In `observeChecks` (observe-status.ts:334), log `deps.log?.warn` with the error message and `.cause` before returning the `"none"` degradation, so the transient failure is auditable.
- Give `GitHubVCS` a logger (constructor-injected, consistent with the composition root) and replace the bare `catch` at vcs.ts:375 with a WARN carrying `codeReviewId` and `.cause` before returning `[]`.
- Replace the direct `checks.listForRef` call at vcs.ts:365 with `this.octokit.paginate(this.octokit.rest.checks.listForRef, { ‚Ä¶ })`, flattening `check_runs` across pages, matching the `.paginate()` pattern already used for pulls/comments/labels in this file.

**Acceptance Criteria**:
- [ ] `observeChecks` regression test: when `getCheckRuns` rejects, `deps.log.warn` is called with the error reason/cause (fails on the pre-fix code, which logs nothing)
- [ ] `getCheckRuns` regression test with a paginated fixture (>100 check runs, the failing run on page 2) returns every run and aggregates to `"failing"`, proving the truncation is gone
- [ ] Build passes after fix

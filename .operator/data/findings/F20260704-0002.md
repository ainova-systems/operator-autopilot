**Severity**: medium
**Priority**: 4
**Files Affected**: 2

**Pattern**: Documentation ↔ code drift in a single-source-of-truth rule
**Domain**: intelligence/rules, packages/core

**Impact**: `intelligence/rules/context.md` describes `packages/core` as "shared types + interfaces, zero runtime code", and `intelligence/rules/typescript.md` states core exports "types, interfaces, error classes (the only runtime)". In reality `packages/core/src/schemas/**` imports and evaluates `zod` at runtime (e.g. `repo.schema.ts`), so core has a real runtime dependency and real runtime exports beyond error classes. The stale claim propagated into three generated files on init PR #1 and was flagged by automated review. Because these rules are the authoritative context loaded into every agent, the drift steers agents toward wrong conclusions (e.g. rejecting legitimate schema code in core, or duplicating schemas elsewhere to "keep core runtime-free").

**Fix**: Update `intelligence/rules/context.md` and `intelligence/rules/typescript.md` (package-boundary table and `@operator/core` sections) to state the permitted runtime precisely: Zod schema values and error classes, `zod` as the single runtime dependency, no cross-workspace runtime imports. Re-sync generated rule outputs. Keep one source: other docs reference the rule instead of restating it.

**Acceptance Criteria**:
- [ ] `intelligence/rules/context.md` and `intelligence/rules/typescript.md` describe core's actual permitted runtime (Zod schemas + error classes)
- [ ] No remaining "zero runtime" claim about `packages/core` anywhere in `docs/**`, `intelligence/**`, or `.operator/**`
- [ ] Build passes after fix

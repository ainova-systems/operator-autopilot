**Pattern**: Scheduling and budget correctness â€” "Backoff counters mixed across causes (lock-skip vs real failure vs empty result); each cause needs its own counter semantics." A real failure gets NO counter semantics at all.
**Domain**: engine/engine/project-runner.ts (per-cycle scheduling for every managed repo)

**Impact**: In `runProject`, the `queue-fill` post-run state update only handles two outcomes:
```
engine/engine/project-runner.ts:113-123
  if (entry.schedule.kind === "queue-fill") {
    if (result.status === "completed") { markScheduleRun(); updateQueueFillBackoff(); }
    else if (result.status === "skipped" && result.message !== "locked") { markScheduleRun(); }
  }
```
There is no branch for `result.status === "failed"`. `safeExecute` (line 238-248) turns any thrown stage error into `{status:"failed"}`, and `runStage` returns `"failed"` for a `failed`/`cancelled` verdict â€” so this path is reachable whenever the queue-fill stage (research â€” an LLM-invoking agent stage; confirmed queue-fill by `qfEntry`/`stateKey:"research"` in the tests) genuinely fails.

Because `markScheduleRun` is never called on failure, the schedule state key is not advanced, so `isScheduleDue`'s base-interval throttle (`state.isScheduleDue(stateKey, intervalMinutes)`, line 213) stays satisfied every subsequent cycle. Because `updateQueueFillBackoff` (line 135-150) is only invoked for completed runs, the exponential-backoff empty-run counter is never bumped either. Net effect: a *persistently failing* research stage re-fires on **every** cycle â€” the exact condition where throttling matters most â€” defeating both the base interval and the exponential backoff and burning agent budget on a broken stage with unbounded frequency. This directly violates the rule that real failures must count against the retry/backoff budget while only lock-skips are exempt; here failures are treated more leniently than empty successes.

The existing tests (`engine/engine/project-runner.test.ts:454-518`) cover completed-reset, completed-bump, locked-skip no-op, and no-eligible-skip throttle-only â€” but there is **no** `failed` case, confirming the gap is untested, not intentional.

**Fix**: Add an explicit `result.status === "failed"` branch in the queue-fill block that advances the schedule (`markScheduleRun`) and bumps the backoff counter (reuse/extend `updateQueueFillBackoff`, or increment `backoffStateKey` directly) so a failing run throttles exactly like an empty one â€” giving the `failed` cause its own explicit counter semantics rather than falling through. Keep the lock-skip no-op untouched. Document the `failed` semantics in the comment block alongside completed/skipped-locked/skipped-other.

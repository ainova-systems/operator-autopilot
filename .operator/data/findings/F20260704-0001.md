**Severity**: medium
**Priority**: 4
**Files Affected**: 2

**Pattern**: Prompt ↔ code drift — a bundled agent prompt asserts engine behavior that the implementing module contradicts
**Domain**: engine/content/prompts/agents

**Impact**: `engine/content/prompts/agents/scout.md` (comment in the generated `project.yaml` template) states the auto-detect order as `CLAUDE.md > AGENTS.md > .cursorrules`, while `engine/config/discovery.ts` implements `AGENTS.md → CLAUDE.md → .cursorrules → .operator/OPERATOR.md`. The drift already propagated into production output: the first self-init PR (#1) shipped a generated `project.yaml` with the wrong order, which had to be caught in human review. Every future onboarding of a managed repo reproduces the error. Related prompts (`context/base.md`, `analyst.md`, `creator.md`) list context files in an order-ambiguous way that invites the same confusion.

**Fix**: Correct the order in `scout.md` to match `detectGlobalContext` in `engine/config/discovery.ts`, including the `.operator/OPERATOR.md` fallback. Align the file listings in `context/base.md`, `analyst.md`, and `creator.md`. Add a contract test that pins the order documented in `scout.md` to the candidate list implemented in `discovery.ts`, so the next drift fails CI instead of shipping.

**Acceptance Criteria**:
- [ ] `scout.md` documents the auto-detect order exactly as implemented in `engine/config/discovery.ts`
- [ ] A test fails if the documented order and the implemented candidate list diverge again
- [ ] Build passes after fix

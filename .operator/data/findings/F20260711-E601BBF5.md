**Severity**: medium
**Priority**: 4
**Files Affected**: 2

**Pattern**: Bounded self-healing, never silent recovery (resilience-boundaries §"Every recovery path … must be bounded by TTL/attempt budget AND logged with what was recovered and why"). This is the exact silent-recovery case the v5 observability mandate forbids (`intelligence/rules/typescript.md` §REQUIRED "every catch-and-continue / fallback path taken MUST warn with the reason"; `migration.md` observability BLOCKER gate).
**Domain**: engine/infra/local (sqlite-guard.ts) — the live `IdempotencyGuard` wired at `engine/entry.ts:133`.

**Impact**: `LocalIdempotencyGuard` is the single lock primitive enforcing "one run per stage per scope" (`docs/workflow.md:600`). It reaps expired locks in two places, both silent:
- `engine/infra/local/sqlite-guard.ts:36` — `DELETE FROM locks WHERE expires_at < ?` bulk-purges every expired row on every `acquire`.
- `engine/infra/local/sqlite-guard.ts:43-50` — when a same-key lock exists but is past `expires_at`, line 49 (`DELETE FROM locks WHERE key = ?`) reaps it and the call proceeds to re-acquire.

The reap is correctly TTL-bounded, but the class constructor (`:25-29`) takes only `dbPath` — it holds no `Logger`, and `acquire` ignores its `_ctx` (`:31`). So when an instance dies mid-stage holding e.g. `stage:task-execute:sample` (ttl 600s) and the next cycle steals the expired lock 10 minutes later, this happens with **zero log output**. The caller of `acquire` only observes success-vs-null; it cannot distinguish "acquired a fresh lock" from "reaped a dead run's lock and re-acquired." A recurring crash loop that repeatedly orphans and reaps stage locks is therefore invisible in the INFO/WARN stream — precisely the "I can't see what the engine did after a failure" gap v5 exists to close. `clearActiveLocks` (`:74`, boot reconciliation) returns a count its caller can log, but the mid-cycle per-`acquire` reap path has no such surface.

**Fix**: Add an optional `readonly log?: Logger` to the `LocalIdempotencyGuard` constructor and thread it from `entry.ts:133`. In `acquire`, before the bulk delete capture the rows that will be reaped (or check `.changes`) and, in the expired-active branch (`:47-49`), emit one WARN naming the reaped `key`, the stolen `lockId`, its `acquired_at`, and how far past `expires_at` it is (e.g. `"reaped expired lock — prior holder likely crashed"`). Keep the reap itself unchanged (already bounded). Log via the injected logger only; preserve the non-throwing behavior.

**Acceptance Criteria**:
- [ ] `LocalIdempotencyGuard` accepts an optional `Logger` (constructor injection), wired from `entry.ts`.
- [ ] `acquire` emits a WARN whenever it reaps an expired lock, including `key`, reaped `lockId`, `acquired_at`, and the overrun past `expires_at`.
- [ ] Failing-first regression test in `sqlite-guard.test.ts` inserts an expired lock row for a key, calls `acquire` for that key, and asserts a WARN naming the reaped key + lockId is logged (fails on pre-fix code where no logger exists), keeping coverage ≥90%.
- [ ] Build passes after fix.

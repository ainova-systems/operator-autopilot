**Severity**: medium
**Priority**: 4
**Files Affected**: 1 (source-of-truth rule doc; evidence across 6 engine source files)

**Pattern**: Docs ↔ code drift — the authoritative engine layer graph understates the real imports of its two lowest layers, contradicting `intelligence/rules/dev-context-engineering.md` ("documentation that drifts from code is a defect; one source per convention").
**Domain**: `intelligence/rules/typescript.md` — "Architecture — Layer graph inside `engine/`"

**Impact**: The layer graph at `intelligence/rules/typescript.md:91-107` is the single source of truth that developers and analyzer agents use to judge whether an intra-engine import is legal. Every other layer that may depend on the base contract lists `@operator/core` explicitly (pipeline/primitives, agents, platforms, storage, communication, events), so the omissions on the last two lines read as deliberate — but they are stale:
- Line 105 `infra/ → Node built-ins only` is false. `engine/infra/env.ts:3` imports `ConfigError` (runtime), `engine/infra/workspace.ts:7` and `engine/infra/workspace-init.ts:6` import `WorkspaceError` (runtime), `engine/infra/instance-heartbeat.ts:4` imports `instanceEntrySchema` (runtime) plus types, and `engine/infra/local/sqlite-guard.ts:2` imports `better-sqlite3` (third-party runtime) — none are Node built-ins. (Line 59 of the same file already notes errors moved out of `engine/infra/errors.ts` into core, which is exactly why infra now depends on core; the graph was never updated.)
- Line 106 `logging/ → nothing` is false. `engine/logging/logger.ts:1` imports `pino` (third-party runtime), `node:stream`, and `type OperationContext` from `@operator/core`.

Because this graph is not ESLint-enforced (the `no-restricted-imports` config in `eslint.operator.config.js` only guards cross-package boundaries, not intra-engine layers), nothing catches the drift automatically. An agent trusting the doc would flag these legitimate core/pino imports as boundary violations, or hesitate to add a needed core import in infra/logging, producing false findings and review churn.

**Fix**: Update the two graph lines in `intelligence/rules/typescript.md` to reflect actual, allowed dependencies, e.g. `infra/ → @operator/core, Node built-ins (+ better-sqlite3 under infra/local/)` and `logging/ → @operator/core (types), pino, Node built-ins`. Keep the source-of-truth file as the only edit; no code change is required since the imports themselves are correct.

**Acceptance Criteria**:
- [ ] `intelligence/rules/typescript.md:105-106` list `@operator/core` (and the third-party runtime deps `better-sqlite3`/`pino`) for `infra/` and `logging/`, matching the actual imports
- [ ] A grep of `engine/infra/**` and `engine/logging/**` non-test imports shows no dependency outside what the updated graph permits
- [ ] Build passes after fix

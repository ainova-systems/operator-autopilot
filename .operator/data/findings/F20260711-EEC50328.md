**Severity**: high
**Priority**: 2
**Files Affected**: 2 (`engine/pipeline/primitives/discovery-selector.ts` root cause + `engine/pipeline/primitives/discovery-selector.test.ts` missing pin)

**Pattern**: Contract change without a cross-boundary pin test (Test Strategy Rules §Regression discipline — "Contract changes without cross-boundary pin tests ... each needs a test exercising the contract from the CONSUMER side"). The CRLF-tolerance frontmatter contract was fixed and regression-tested on ONE parser and left unpinned on a parallel parser that reads the same on-disk frontmatter shape.
**Domain**: engine/pipeline/primitives (discovery-selector) — the research-stage analyzer loader

**Impact**: Commit `700dbf6` fixed the CRLF frontmatter bug in `engine/work-items/work-items.ts:parseFrontmatterFields` (`block.split("\n")` → `block.split(/\r?\n/)`) and pinned it with a named regression test (`work-items.test.ts:164` "parses CRLF frontmatter ... (regression)"). The identical frontmatter contract is parsed a second time by `parseAnalyzerFrontmatter` (`engine/pipeline/primitives/discovery-selector.ts:81-90`), which STILL uses `block.split("\n")` with the same no-`m`-flag `/^(\w[\w_]*):\s*(.*)$/` regex — the exact pre-fix logic, self-labelled "Keeps semantics of v4 `parseFrontmatterFields`" (`:80`). Because `(.*)$` (no `m` flag) cannot match before the trailing `\r`, every field silently fails to parse on a CRLF checkout — the precise failure mode the work-items fix comment documents ("managed-repo files checked out on Windows / with git autocrlf carry `\r\n` ... every field silently fails to parse"). Consequence in `loadAnalyzerDefs` (`:118-125`): `schedule` falls back to `"daily"` (a `weekly`/`on-demand` analyzer runs EVERY cycle → wasted agent spend + duplicate findings), `enabled` falls back to `true` (an `enabled: false` analyzer RUNS), and `path` becomes `undefined` (analyzer scope scans the whole repo). This is a primitive (>=95% coverage bar), yet `discovery-selector.test.ts` (287 lines) contains no CRLF fixture — every fixture, including the `enabled: false` case at `:126`, uses `\n` only, so the contract is unpinned on the consumer side and a real, already-encountered bug class survives.

**Fix**: Add a failing-first CRLF regression test to `discovery-selector.test.ts` that writes an analyzer file with `\r\n` line endings and asserts `enabled: false` stays `false`, `schedule: weekly:3` stays throttled, and `path` is preserved — it must fail on the current `split("\n")` code. Then align `parseAnalyzerFrontmatter` with the work-items fix (`block.split(/\r?\n/)`), mirroring the same quote-stripping tolerance. Consider extracting the single shared frontmatter-field parser so the contract has one pinned implementation instead of three drifting copies.

**Acceptance Criteria**:
- [ ] `discovery-selector.test.ts` gains a test named for the CRLF scenario (e.g. "parses CRLF (\r\n) analyzer frontmatter — enabled:false stays disabled and weekly stays throttled") that fails on the pre-fix `split("\n")` parser and passes after the fix
- [ ] `parseAnalyzerFrontmatter` splits on `/\r?\n/` so `enabled`, `schedule`, and `path` parse identically under LF and CRLF
- [ ] Coverage on `discovery-selector.ts` stays >=95% with the new edge-path test
- [ ] Build passes after fix

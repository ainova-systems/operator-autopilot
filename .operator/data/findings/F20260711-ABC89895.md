**Severity**: high
**Priority**: 2
**Files Affected**: 3

**Pattern**: Agent output contract consistency — verdict vocabulary (P2). The prompt actually sent to the verifier LLM under-specifies the verdict vocabulary relative to what the parser accepts and what the bundled `verifier.md` contract requires, and the authoritative `verifier.md` is never loaded into the review call.
**Domain**: engine/agents (verifier review path) + engine/content/prompts/agents/verifier.md

**Impact**: The verifier's real prompt is built inline by `buildReviewPrompt` (`engine/agents/runtime.ts:534-541`), whose only output instruction is: `"Respond with APPROVED if all criteria met, or REJECTED: <specific issues>."` — two verdicts, no `## Verdict:` marker. But:
- `parseReviewVerdict` (`engine/agents/runtime.ts:566-577`) prefers `## Verdict: (APPROVED|RETRY|FAILED|CANCELLED|REJECTED)` and its own comment states this "is the structured format the new verifier prompt asks for" — the prompt does NOT ask for it.
- The bundled `verifier.md` fully documents the five-verdict contract (`verifier.md:17` "You must return exactly one of five verdicts") and the `## Verdict: <KIND>` output format (`verifier.md:99-151`), yet `runReview` calls `verifier.execute(reviewPrompt, { model, timeoutMs, cwd })` (`engine/agents/runtime.ts:513-518`) with NO `systemPromptFile`, so `agents/verifier.md` (the verifier role's `instructions` in `engine/content/defaults/agents.yaml`) is never sent to the model.

Because the LLM is only told to emit APPROVED or bare `REJECTED`, and bare `REJECTED` is mapped to `retry` (`engine/agents/runtime.ts:595-598`), the terminal `FAILED`/`CANCELLED`/`REJECTED` dispositions are effectively unreachable. A task whose premise is invalid (target file gone, already fixed) — which `verifier.md` says must be `CANCELLED`/`REJECTED` to close the PR and terminate the work item — instead loops through the full retry budget and only then surfaces as exhausted. The elaborate disposition logic in `verifier.md` is dead, and the parser's `## Verdict:` branch is never exercised on live output.

**Fix**: Fix the code side (the review prompt), which is the side that disagrees with both the parser and `verifier.md`. Either (preferred) have `buildReviewPrompt` emit the documented five-verdict instruction — require a final `## Verdict: <APPROVED|RETRY|FAILED|CANCELLED|REJECTED>` block with the same guidance `verifier.md` gives — or load `verifier.md` as the verifier's system prompt for the review call (pass its resolved path as `systemPromptFile` to `verifier.execute`) so the bundled contract is actually delivered. Keep `parseReviewVerdict` as-is; it already accepts the full vocabulary.

**Acceptance Criteria**:
- [ ] The prompt the verifier LLM receives instructs it to output all five verdicts using the `## Verdict: <KIND>` format that `parseReviewVerdict` recognizes (either via `buildReviewPrompt` text or by loading `verifier.md`).
- [ ] A contract test pins the two sides together: it asserts every verdict literal accepted by `parseReviewVerdict`'s `## Verdict:` regex (APPROVED, RETRY, FAILED, CANCELLED, REJECTED) is named in the review prompt the verifier actually receives; it fails on the current pre-fix prompt.
- [ ] Either `agents/verifier.md` is loaded into the review call, or its role `instructions` entry in `agents.yaml` is reconciled so no bundled prompt file is silently orphaned.
- [ ] Build passes after fix.

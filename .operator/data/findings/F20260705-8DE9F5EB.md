**Severity**: high
**Priority**: 2
**Files Affected**: 1 (prompt is the wrong side; evidence spans parser + wiring)

**Pattern**: AOP EMIT contract drift — a bundled prompt instructs the agent to emit AOP records in a marker syntax the parser does not accept, so 100% of the improver's emitted lifecycle records are silently discarded on a successful run.
**Domain**: engine/content/prompts/agents/improver.md (retrospective / weekly-metrics stage)

**Impact**: The AOP text-block parser recognizes ONLY fenced markers — `engine/pipeline/primitives/agent-output-protocol.ts:41` `START_MARKER = /^===\s*EMIT\s+([a-zA-Z][a-zA-Z0-9-]*)\s*===\s*$/` and `:42` `END_MARKER = /^===\s*END EMIT\s*===\s*$/`. Every other prompt obeys this: `supervisor.md` uses 14 `=== EMIT ... ===` blocks, `planner.md` and `analyst.md` 3 each. The improver prompt is the sole exception — it has **zero** fenced markers and instead documents a YAML-key pseudo-syntax:
```
engine/content/prompts/agents/improver.md:100-105
  ```yaml
  EMIT: status-update
  target: F20260416-0002
  status: in-progress
  reason: "children ... already merged on develop; parent never advanced"
  ```
engine/content/prompts/agents/improver.md:111-112
  child items (`EMIT: child-item`) and rewriting work-item bodies (`EMIT: body-update`).
```
The improver IS wired through this parser: `engine/engine/stage-handlers.ts:262` builds `new TextBlockEventStream(parseAgentOutput)` (the fenced parser) and `engine/pipeline/composers/weekly-metrics-stage.ts:300` feeds the improver's stdout to `applyAgentEvents` via that stream. A line like `EMIT: status-update` never matches `START_MARKER`, is treated as free text, and — lacking a `---` fence — does not even trip the `raw-frontmatter-leak` guard (`agent-output-protocol.ts:90-103`). Result: `parseResult.events` is empty, `applyAgentEvents` returns `statusUpdates: [], childItems: []`, and `weekly-metrics-stage.ts:311` logs nothing — a silent no-op. This guts the improver's core lifecycle mechanism: Goal 2 task cancellation, Goal 3 duplicate detection, and every "Status Reconciliation" the improver reports (`improver.md:65-113`, `:135-136`) are emitted-but-dropped. The prompt even forbids the only working fallback — "never bypass it by editing frontmatter directly" (`improver.md:109-110`) — so the improver has NO functioning path to reconcile status, while its report claims it did.

**Fix**: Rewrite the three `EMIT:` occurrences in `improver.md` to the fenced form the parser accepts, matching `supervisor.md:70-75` / `planner.md:86-107`, e.g.:
```
=== EMIT status-update ===
target: F20260416-0002
status: in-progress
reason: "..."
=== END EMIT ===
```
and reference `EMIT child-item` / `EMIT body-update` as fenced blocks in the prose. Code is correct; only the prompt is wrong.

**Acceptance Criteria**:
- [ ] `improver.md` contains no `EMIT:` colon-syntax; its status-update/child-item/body-update examples use `=== EMIT <type> === ... === END EMIT ===`
- [ ] A contract test feeds the improver.md status-update example block through `parseAgentOutput` and asserts it yields one `status-update` `AgentEvent` with zero error diagnostics (fails on the pre-fix `EMIT:` text, passes after)
- [ ] Build passes after fix (`npm run typecheck && npm run lint && npm test`)

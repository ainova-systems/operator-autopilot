**Severity**: medium
**Priority**: 4
**Files Affected**: 1

**Pattern**: Expected-outcome API probe surfaces on the ERROR stream — observability noise that erodes trust in ERROR
**Domain**: engine/platforms/github

**Impact**: During normal label transitions on PR #1 the cycle log showed `ERROR GET /repos/.../labels/ai%3Aprocessing - 404` and `ERROR GET /repos/.../labels/ai%3Ain-review - 404`, immediately followed by successful label creation. The 404 is the expected "label does not exist yet, create it" branch of the ensure-label flow, yet it is emitted at ERROR level (Octokit request-error logging). The v5 observability tenet says every ERROR line must represent a real failure; expected probes at ERROR level train operators to ignore the ERROR stream, which is how real failures get missed.

**Fix**: In the ensure-label (get-then-create) flow, treat the existence-probe 404 as an expected branch: log at DEBUG with the "label missing, creating" reason, and keep ERROR only for non-404 failures or for 404s outside existence probes. Audit other existence probes (branch, file, PR lookups) for the same pattern.

**Acceptance Criteria**:
- [ ] A full label-bootstrap cycle (label absent → created) produces zero ERROR-level lines
- [ ] Non-404 label API failures still log at ERROR with full cause
- [ ] A regression test pins the ensure-label happy path to no-error-level logging
- [ ] Build passes after fix
